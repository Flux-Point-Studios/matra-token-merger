PRE-LAUNCH VALIDATION

1. Overview

This document describes the three-level mainnet validation procedure for the cMATRA Token Merger Portal. Each level increases in risk and realism, allowing the team to build confidence that the full surrender pipeline works correctly on mainnet before opening the redemption window to the public.

Level Test Risk What It Proves
1 Dry-run evaluate_tx Zero Plutus script executes against live mainnet UTxOs. Correct script hash, valid parameters, fees, UTxO references.
2 Admin canary surrender Minimal Full end-to-end pipeline on mainnet: build, co-sign, submit, on-chain confirmation. Verifies cMATRA delivery, legacy quarantine, pool remainder.
3 Portal wallet flow Minimal Full user experience: browser portal, CIP-30 wallet connect, scan, confirmation dialog, wallet signature, on-chain submission.

Principle: Each level must pass before advancing to the next. Level 1 failure means "do not touch mainnet." Level 2 failure means "do not open the portal." Level 3 success means "ready for public launch."

3. Level 1 — Dry-Run Evaluation

ZERO RISK No transaction is submitted to the network.

3.1 What It Does

Builds a complete surrender transaction using real mainnet UTxOs (the funded pool, the user's address), then submits the serialized transaction to Blockfrost's evaluate_tx endpoint. This endpoint runs the Plutus V3 script against the live ledger state and returns execution units (CPU + memory) or a detailed error if the script fails. The transaction is never broadcast to the network.

3.2 Procedure

# 1. Start the surrender API service
py -3.12 -m services.surrender_api

# 2. In a second terminal, run the canary script (evaluate only)
py -3.12 scripts/mainnet_canary.py \
  --address addr1<your_admin_wallet_address> \
  --asset AGENT \
  --qty 1

3.4 Pass / Fail Criteria

OutcomeMeaningAction
PASSED Script evaluation returned execution units Proceed to Level 2
FAILED Script evaluation returned an error Do NOT proceed. Diagnose: wrong script params? Missing datum? Incorrect pool UTxO format? Fix and re-run.

4. Level 2 — Admin Canary Surrender

MINIMAL RISK A real transaction is submitted. Uses a trivially small amount.

4.1 What It Does

Performs a real surrender of a tiny amount of legacy assets from the admin's own wallet. The admin key acts as both the service signer and the user signer (self-test). The transaction is submitted to the Cardano mainnet. After on-chain confirmation, the result is verified: cMATRA received, legacy asset quarantined, pool remainder returned.

Cost: Surrendering 1 AGENT costs approximately 0.35 ADA in fees and permanently burns 1 AGENT token (worth fractions of a cent). The admin receives ~0.54 cMATRA in return. This is a real, irreversible transaction.

4.2 Procedure

# Run with --submit flag (prompts for confirmation)
py -3.12 scripts/mainnet_canary.py \
  --address addr1<your_admin_wallet_address> \
  --asset AGENT \
  --qty 1 \
  --submit

4.3 Post-Submission Verification

After the script reports success, verify on Cardanoscan:

  1. cMATRA output — The admin address received the correct cMATRA amount
  2. Quarantine output — 1 AGENT was sent to the quarantine address
  3. Pool remainder — The remaining pool balance was returned to the script address with a Void datum
  4. Fee — Transaction fee is reasonable (typically 0.3-0.5 ADA for a Plutus tx)

4.4 Pass / Fail Criteria

OutcomeMeaningAction
PASSED All four verification points confirmed on-chain Proceed to Level 3
FAILED Submission rejected or outputs incorrect Do NOT proceed. Investigate the on-chain result or error. The pool UTxO may need to be re-evaluated.

5. Level 3 — Portal Wallet Flow

MINIMAL RISK Full user-facing flow through the web portal with a real CIP-30 wallet.

5.1 What It Does

A team member (admin or trusted tester) opens the merger portal in a browser, connects a Cardano wallet (Eternl, Lace, Nami, etc.), and completes a surrender through the full user interface. This validates the entire stack end-to-end: frontend rendering, wallet integration, server-side scanning, co-signing flow, wallet signature prompt, and on-chain submission.

5.2 Procedure

  1. Navigate to fluxpointstudios.com/matra-merger
  2. Enter the portal access password
  3. Click "Connect Wallet" and select your Cardano wallet
  4. Verify the scan results: correct assets and cMATRA amounts displayed
  5. Click "Surrender All & Receive cMATRA"
  6. Review the confirmation dialog — verify it shows the correct total and irreversibility warning
  7. Click "Confirm Surrender"
  8. Approve the transaction in your wallet (review inputs/outputs in the wallet's tx preview)
  9. Wait for on-chain confirmation
  10. Verify the success state: tx hash link displayed, cMATRA in wallet
